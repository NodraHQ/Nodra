01

Introduction

A blockchain is a distributed ledger: a database that is copied and synchronized across many independent computers, called nodes, instead of being stored on a single server controlled by one entity. Every participant holds a copy of the same history, and that history is agreed upon collectively rather than dictated by a central authority.

The term "blockchain" describes the structure of that ledger: data is grouped into blocks, and each block is cryptographically linked to the one before it, forming a chain. Once a block is added and confirmed by the network, altering it would require rewriting every block that came after it on every copy of the ledger at the same time — which is what makes the record practically immutable.

This lesson walks through the core mechanics behind that idea: how blocks are built, how nodes agree on which version of the ledger is correct, how the network is organized into layers, and how users interact with it safely through wallets.

02

How Blockchain Works

At a high level, a blockchain network performs four repeating actions: transactions are broadcast, nodes validate them, valid transactions are grouped into a block, and the network agrees on adding that block to the chain. The four concepts below are the building blocks of that cycle.

◧

Decentralization

No single computer owns the ledger. Thousands of independent nodes each keep a full or partial copy, so there is no single point of failure or control.

◨

Transactions

A transaction is a signed instruction, such as "send 0.5 units from address A to address B." It is broadcast to the network and waits to be picked up by nodes for validation.

◩

Validation

Nodes check that a transaction follows the protocol's rules — correct signature, sufficient balance, no double-spending — before it can be included in a block.

◪

Immutability

Once a block is confirmed and buried under later blocks, changing it becomes computationally and economically impractical, giving the ledger its permanent, tamper-evident nature.

03

Anatomy of a Block

Every block is made of a header and a body. The header contains metadata used to link and verify the block; the body contains the actual transaction data.

block_847213.json
Block Number #847,213
Timestamp 2026-06-14T10:32:05Z
Previous Hash 0x7c2e8f1a2b3d4e5f...
Merkle Root 0x9b1c4d7e0f2a3b5c...
Nonce 2,847,193,022
Block Hash 0x1d8b6a4c8e2f9a1d...
Transactions 1,942 transactions
  • Previous Hash — links this block to the one before it, forming the chain.
  • Merkle Root — a single hash that summarizes every transaction in the block, allowing quick verification without re-checking each one individually.
  • Nonce — a number miners adjust repeatedly while searching for a valid block hash under Proof of Work.
  • Block Hash — the fingerprint of the entire block header, recalculated by every node to confirm the block is valid.

Hash Demo

A cryptographic hash function takes any input and returns a fixed-length string of characters. The same input always produces the same output, but even a one-character change produces a completely different result. This is what makes tampering with a block detectable.

Notice the second input only changes the capitalization of one letter, yet the resulting hash is entirely different from the first. This property is called the avalanche effect, and it is what allows nodes to instantly detect if a block's contents have been altered. 05

Consensus

Because there is no central authority, the network needs a way for independent nodes to agree on a single, shared version of the ledger. This agreement mechanism is called a consensus algorithm. The two most widely used models are Proof of Work and Proof of Stake.

Proof of Work (PoW)

Nodes called miners compete to solve a computationally expensive puzzle: finding a nonce that, combined with the block's data, produces a hash below a target value. The first miner to find a valid solution broadcasts the block, and other nodes verify it instantly. Solving the puzzle is hard; checking the solution is easy.

  • Used by Bitcoin and (historically) Ethereum.
  • Security comes from the cost of computing power (energy and hardware).
  • High energy consumption is its main criticism.

Proof of Stake (PoS)

Instead of competing with computing power, validators lock up ("stake") an amount of the network's native asset. The protocol selects a validator to propose the next block, generally weighted by stake size. Validators who approve invalid blocks can lose part of their stake, a penalty known as slashing.

  • Used by Ethereum (since the Merge), Cardano, and Solana.
  • Security comes from the economic value at stake, not raw computation.
  • Dramatically lower energy consumption than PoW.
06

Layer 1 vs Layer 2

As blockchains grew in usage, a single base network struggled to process every transaction quickly and cheaply. This led to a layered architecture, where additional networks are built on top of the base chain to scale it.

Layer 1

The base blockchain itself — Bitcoin, Ethereum, Solana. It handles consensus, security, and final settlement directly. Changes to its core rules require broad network agreement and are made carefully, since everything else depends on this foundation.

Layer 2

A separate protocol built on top of a Layer 1 that processes transactions off the base chain and later settles a summary back onto it. Examples include rollups and state channels. This increases throughput and lowers fees while still inheriting the base layer's security.

07

Wallets

A wallet does not "store" coins the way a physical wallet stores cash. It stores the private keys that prove ownership of assets recorded on the blockchain, and uses those keys to sign transactions.

Hot Wallet

Connected to the internet at all times — mobile apps, browser extensions, exchange accounts. Convenient for frequent transactions, but more exposed to malware, phishing, and remote attacks.

Cold Wallet

Keeps private keys offline — hardware devices or paper backups. Transactions are signed in an isolated environment, which makes cold wallets far more resistant to remote hacking, at the cost of convenience for everyday use.

A common practice is to keep small, frequently used amounts in a hot wallet and larger holdings in a cold wallet — the same logic as keeping cash in a pocket versus savings in a bank vault.

08

Security

Blockchain security is a mix of cryptography, economic incentives, and network design. Understanding where the guarantees actually come from helps separate real risk from perceived risk.

  • Private keys are the real target. Most losses come from stolen or leaked keys, not from breaking the blockchain's cryptography.
  • 51% attacks. If a single actor controls the majority of a network's mining power or stake, they could theoretically rewrite recent history — which is why decentralization of that power matters.
  • Smart contract risk. Code deployed on-chain can contain bugs; audits reduce but do not eliminate this risk.
  • Phishing and social engineering. The most common attack vector targets the user directly, not the protocol.

First Steps

A practical path for putting this lesson into practice, from observation to safe first interaction.

  1. 1

    Explore a block explorer

    Open a public block explorer for any major chain and inspect a real block: its hash, its transactions, its timestamp.

10

Glossary

Node
A computer participating in the network, holding a copy of the ledger and enforcing its rules.
Hash
A fixed-length output produced by feeding data into a cryptographic hash function.
Nonce
A number that miners vary to find a valid block hash under Proof of Work.
Merkle Root
A single hash summarizing all transactions in a block, used for quick verification.
Validator / Miner
A network participant responsible for proposing and confirming new blocks.
Private Key
A secret value that proves ownership of an address and is used to sign transactions.
Rollup
A Layer 2 solution that executes transactions off-chain and posts compressed results back to Layer 1.
Double-Spending
Attempting to spend the same digital asset more than once, prevented by network validation.
